Upcoming Contests API!

Easy to use REST API for fetching upcoming contests. WITHOUT any request throttle.

Useful features like server-side pagination and filters.

Documenation

Data Format

Filters

Platforms

Datatype : Array of Strings

This filter will return only the contests of the specified platforms

start_gte

Datatype: Number

Description: Time in milliseconds

This filter will return the contests whose starting time is GREATER THAN EQUAL to the specified time

start_lte

Datatype: Number

Description: Time in milliseconds

This filter will return the contests whose starting time is LESS THAN EQUAL to the specified time

end_gte

Datatype: Number

Description: Time in milliseconds

This filter will return the contests whose ending time is GREATER THAN EQUAL to the specified time

end_lte

Datatype: Number

Description: Time in milliseconds

This filter will return the contests whose ending time is LESS THAN EQUAL to the specified time

duration_gte

Datatype: Number

Description: Time in milliseconds

This filter will return the contests whose duration is GREATER THAN EQUAL to the specified time

duration_lte

Datatype: Number

Description: Time in milliseconds

This filter will return the contests whose duration is LESS THAN EQUAL to the specified time

pagenumber

Datatype: Number

This is for Server Side Pagination. This will filter the data and only provide documents which will be displayed on the specified page.

itemsperpage

Datatype: Number

This is for Server Side Pagination. This will specify how many documents to display on a page.

More Info

  • There is no limit parameter in the API. You can simulate limit parameter by passing pagenumber as 1 as itemsperpage as limit.
  • The Database is updated everyday at 1AM. Request will be returned with Server error from 00:58 AM till the time database is updated to prevent any deadlocks. This usually takes a few seconds.
  • The data for this API is taken from clist.by. UC-API is much better alternative to clist.by as
    • clist.by data structure is complex.
    • faster that clist.by
    • clist.by has a throttle of only 10 requests per minute! Which makes it non-scalable!
